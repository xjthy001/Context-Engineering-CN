# 文档协议

> *"写作就是思考。写得好就是思考清楚。这就是为什么它如此困难。"*
>
> **— 大卫·麦卡洛 (David McCullough)**

## 文档协议简介

文档协议将混乱的内容创作过程转变为结构化、高效的工作流,持续产出高质量结果。它们为组织想法、管理信息和制作引人注目的内容提供架构框架——同时优化您与AI系统的交互。

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│              文档协议的优势                          │
│                                                     │
│  • 一致的文档质量和结构                              │
│  • 减少创作过程中的认知开销                          │
│  • 人类和AI之间的高效协作                            │
│  • 从概念到完成的清晰进展                            │
│  • 复杂文档的优化token使用                           │
│  • 内容随时间可维护和可演化                          │
│                                                     │
└─────────────────────────────────────────────────────┘
```

本指南为常见的内容创作场景提供可立即使用的文档协议,配有实施指导和性能指标。每个协议都遵循我们的NOCODE原则:导航(Navigate)、编排(Orchestrate)、控制(Control)、优化(Optimize)、部署(Deploy)和演化(Evolve)。

## 如何使用本指南

1. **选择协议**,匹配您的文档创作目标
2. **复制协议模板**并自定义占位符
3. **在交互开始时向AI助手提供协议**
4. **遵循结构化过程**,从初始概念到最终文档
5. **监控指标**以评估有效性
6. **迭代和优化**您的协议以供未来使用

**苏格拉底式提问**:您发现最难创建的文档类型是什么?文档创作过程的哪些方面消耗了最多的时间和精神能量?

## 1. 结构化文章协议

**何时使用此协议:**
需要创建全面、结构良好的文章,有效传达复杂信息?此协议指导您开发具有清晰组织、平衡深度和引人入胜内容的文章——非常适合博客文章、教育内容、思想领导力作品或技术解释。

```
/document.article{
    intent="创建关于特定主题的全面、结构良好的文章",
    input={
        topic="[文章主题]",
        target_audience="[主要读者及其知识水平]",
        key_points=["[要点_1]", "[要点_2]", "[要点_3]", "[根据需要添加]"],
        tone="[期望语调: 学术/对话/说服等]",
        length="[大约字数]",
        special_elements="[任何特定包含内容: 示例/案例研究/数据等]"
    },
    process=[
        /outline{
            action="创建详细的层次结构",
            format="包含章节和子章节的大纲"
        },
        /develop{
            action="将大纲扩展为完整内容",
            sections=[
                /introduction{
                    elements=["钩子", "背景", "论点", "路线图"],
                    purpose="吸引读者并建立框架"
                },
                /body{
                    approach="想法的逻辑进展",
                    section_pattern=[
                        "关键点陈述",
                        "支持证据/解释",
                        "说明性示例",
                        "影响或应用"
                    ]
                },
                /conclusion{
                    elements=["摘要", "更广泛的背景", "行动号召/未来方向"],
                    purpose="强化关键信息并提供闭合"
                }
            ]
        },
        /enhance{
            elements=[
                "章节间的过渡短语",
                "多样化的句子结构",
                "精确而引人入胜的词汇",
                "适合语调的修辞手法"
            ]
        },
        /finalize{
            action="审查和优化完整文章",
            checks=["清晰度", "流畅性", "语调一致性", "证据力度"]
        }
    ],
    output={
        final_article="具有清晰结构和引人入胜内容的完整文章",
        key_messages="文章中提出的中心点摘要",
        suggested_title="推荐标题和可能的替代方案",
        outline_reference="最终结构供未来参考"
    }
}

我想按照此协议使用我提供的信息创建一篇文章。请确认并从大纲开始。
```

### 实施指南

1. **主题定义**:
   - 要具体而非笼统
   - 框定为集中的问题或陈述
   - 考虑广度(覆盖范围)和深度(详细程度)

2. **受众规范**:
   - 定义人口统计、知识水平和兴趣
   - 考虑他们阅读此内容的动机
   - 识别他们从文章中寻求的价值

3. **关键点选择**:
   - 识别3-7个核心信息或论点
   - 确保点之间逻辑构建
   - 平衡覆盖广度与有意义的深度

4. **语调设置**:
   - 使语调与受众和目的匹配
   - 考虑适当的词汇和句子结构
   - 在整个文档中保持一致性

### 性能指标

| 指标 | 描述 | 目标 |
|------|-------------|--------|
| 结构完整性 | 逻辑组织和流畅性 | 清晰的层次结构和平滑过渡 |
| 内容深度 | 要点发展的彻底性 | 每个关键点的实质性探索 |
| 参与价值 | 读者兴趣和保留 | 引人注目的钩子和多样化的节奏 |
| 信息清晰度 | 关键点的可理解性 | 不可错认的中心信息 |

### 应用示例

```
/document.article{
    intent="创建关于特定主题的全面、结构良好的文章",
    input={
        topic="生成式AI对内容创作工作流的影响",
        target_audience="具有基本AI知识但技术专业知识有限的营销专业人士",
        key_points=[
            "AI从根本上改变内容创作经济学",
            "人机协作需要新的工作流和技能",
            "质量控制比初始创作更重要",
            "伦理考虑应融入新流程"
        ],
        tone="信息丰富但对话式,前瞻但实用",
        length="1200-1500字",
        special_elements="包括实际示例、小型案例研究和可操作的要点"
    },
    process=[...],
    output={...}
}
```

## 2. 技术文档协议

**何时使用此协议:**
创建需要清晰、准确和全面的技术文档?此协议构建技术指南、API文档、产品手册或流程文档的开发,重点关注可用性、技术准确性和适当的详细级别。

```
/document.technical{
    intent="创建精确、可用的技术文档",
    input={
        subject="[要记录的系统/产品/流程]",
        documentation_type="[指南/参考/API文档/手册等]",
        target_users="[主要用户及其专业水平]",
        key_components=["[组件_1]", "[组件_2]", "[组件_3]", "[根据需要添加]"],
        technical_depth="[基础/中级/高级]",
        usage_context="[文档将如何及在何处使用]"
    },
    process=[
        /structure{
            action="创建逻辑文档架构",
            elements=[
                "概述部分",
                "先决条件信息",
                "逐个组件的详细信息",
                "交叉引用和关系",
                "故障排除或常见问题部分"
            ]
        },
        /develop{
            action="以适当的详细程度构建技术内容",
            sections=[
                /overview{
                    elements=["目的", "范围", "架构", "关键概念"],
                    purpose="提供背景和高层理解"
                },
                /component_details{
                    for_each="key_component",
                    structure=[
                        "组件目的和背景",
                        "技术规格",
                        "使用说明或示例",
                        "限制和考虑因素"
                    ]
                },
                /cross_references{
                    action="建立组件之间的连接",
                    elements=["依赖关系", "交互", "工作流"]
                },
                /troubleshooting{
                    structure=["常见问题", "诊断步骤", "解决方案"],
                    purpose="实现自助问题解决"
                }
            ]
        },
        /enhance{
            elements=[
                "清晰的图表或视觉辅助",
                "代码示例或配置示例",
                "重要信息的标注",
                "一致的术语和定义"
            ]
        },
        /finalize{
            action="审查和验证技术准确性",
            checks=["正确性", "完整性", "可用性", "可访问性"]
        }
    ],
    output={
        final_documentation="具有适当结构和详细信息的完整技术文档",
        terminology_glossary="关键技术术语的定义",
        usage_examples="展示应用的实际示例",
        improvement_areas="未来文档增强的建议"
    }
}

我想按照此协议使用我提供的信息创建技术文档。请确认并从文档结构开始。
```

### 实施指南

1. **主题定义**:
   - 清楚地定义范围和边界
   - 识别要记录的特定版本或迭代
   - 考虑系统架构和组件关系

2. **文档类型选择**:
   - 指南:流程的分步说明
   - 参考:用于查找的全面信息
   - API文档:接口规格和使用
   - 手册:完整的产品或系统操作

3. **用户规范**:
   - 定义主要用户的技术专业水平
   - 考虑具有不同需求的次要用户群
   - 识别常见使用场景和用户目标

4. **组件优先级排序**:
   - 列出主要功能区域或系统组件
   - 根据重要性和使用频率确定优先级
   - 考虑逻辑分组和关系

### 性能指标

| 指标 | 描述 | 目标 |
|------|-------------|--------|
| 技术准确性 | 所有信息的正确性 | 100%准确性 |
| 完整性 | 所有必要组件的覆盖 | 无功能空白 |
| 可用性 | 查找和应用信息的便利性 | 2-3次点击/步骤内可访问信息 |
| 示例质量 | 示例的实用性和清晰度 | 直接适用于常见场景 |

### 应用示例

```
/document.technical{
    intent="创建精确、可用的技术文档",
    input={
        subject="客户数据管理系统v2.1的RESTful API",
        documentation_type="带使用指南的API参考",
        target_users="具有中级REST经验的后端开发人员,一些新接触我们特定实现",
        key_components=[
            "身份验证和授权",
            "客户记录端点",
            "交易历史端点",
            "批量操作",
            "速率限制和配额"
        ],
        technical_depth="中级带高级部分",
        usage_context="在开发实施期间使用,用于故障排除"
    },
    process=[...],
    output={...}
}
```

## 3. 高管简报协议

**何时使用此协议:**
需要向忙碌的决策者简明地传递复杂信息?此协议创建专注的执行摘要、简报或备忘录,高效地传递必要信息,同时保持影响力和可操作性。

```
/document.executive_brief{
    intent="为决策者创建简洁、有影响力的简报",
    input={
        topic="[简报主题]",
        key_points=["[要点_1]", "[要点_2]", "[要点_3]", "[根据需要添加]"],
        decision_context="[此简报支持的决策或行动]",
        audience="[特定决策者及其优先事项]",
        data_elements="[要包括的关键数据点]",
        length_constraint="[最大长度或阅读时间]"
    },
    process=[
        /prioritize{
            action="根据决策相关性识别和排列信息",
            criteria=["对决策的影响", "紧迫性", "战略一致性"]
        },
        /structure{
            action="创建适合高管的格式",
            elements=[
                "标题摘要(关键信息)",
                "背景(最少必要背景)",
                "洞察(关键发现和影响)",
                "建议(清晰、可操作的后续步骤)",
                "支持证据(选定的高影响数据)"
            ]
        },
        /develop{
            action="制作简洁、精确的内容",
            approach=[
                "使用高管词汇和语调",
                "强调洞察而非过程",
                "关注业务影响",
                "保持简洁而不牺牲清晰度"
            ]
        },
        /enhance{
            elements=[
                "视觉数据呈现",
                "问题的高管框架",
                "清晰的决策路径",
                "风险和机会评估"
            ]
        },
        /finalize{
            action="优化即时理解",
            checks=["清晰度", "可操作性", "战略一致性", "简洁性"]
        }
    ],
    output={
        executive_brief="为决策者消费优化的简洁文档",
        key_takeaways="单句核心信息",
        recommended_actions="优先行动项",
        supporting_data="选定的高影响证据点"
    }
}

我想按照此协议使用我提供的信息创建高管简报。请确认并从优先处理关键信息开始。
```

### 实施指南

1. **主题框架**:
   - 以业务影响或战略相关性的术语表达
   - 专注于结果而非过程
   - 以决策者的语言框架,而非技术术语

2. **决策背景阐明**:
   - 指定此简报将告知的确切决策
   - 注意决策的时间线
   - 识别影响决策的约束或考虑因素

3. **受众分析**:
   - 定义具体角色和责任
   - 注意每个决策者的特定关注或优先事项
   - 考虑现有知识和偏好

4. **数据选择**:
   - 只选择最有影响力的指标或发现
   - 专注于前瞻性影响而非历史细节
   - 以业务术语呈现数据(收入、成本、增长、风险)

### 性能指标

| 指标 | 描述 | 目标 |
|------|-------------|--------|
| 简洁性 | 沟通效率 | 目标时间内可读(通常3-5分钟) |
| 高管相关性 | 与决策者优先事项的一致性 | 与战略关注的直接联系 |
| 可操作性 | 后续步骤的清晰度 | 明确的建议 |
| 影响清晰度 | 重要性的明显性 | 清晰的业务影响 |

### 应用示例

```
/document.executive_brief{
    intent="为决策者创建简洁、有影响力的简报",
    input={
        topic="拟议扩展到东南亚市场:机会分析和进入策略",
        key_points=[
            "泰国和越南提供最高的即时投资回报率,预计利润率为28%",
            "分阶段进入策略与同步发布相比减少40%的资本需求",

            "竞争格局显示在主要竞争对手进入之前有12-18个月的窗口"
        ],
        decision_context="董事会需要在下次会议上批准1500万美元的初始投资和合作伙伴协议",
        audience="具有不同国际经验的董事会成员,特别关注风险管理和资本效率",
        data_elements="市场规模预测、竞争对手分析、各阶段资本需求、风险评估矩阵",
        length_constraint="最多5分钟阅读,一页执行摘要"
    },
    process=[...],
    output={...}
}
```

## 4. 教学内容协议

**何时使用此协议:**
创建旨在教授技能或程序的内容?此协议开发教育材料,重点关注学习进展、知识保留和实际应用——非常适合教程、操作指南、培训材料或教育内容。

```
/document.instructional{
    intent="创建有效的以学习为重点的内容",
    input={
        subject="[要教授的技能或知识]",
        learner_profile="[目标学习者及其起点]",
        learning_objectives=["[目标_1]", "[目标_2]", "[目标_3]", "[根据需要添加]"],
        prerequisite_knowledge="[学习者应该已经知道的内容]",
        format="[教程/指南/课程等]",
        engagement_approach="[如何保持学习者兴趣]"
    },
    process=[
        /structure{
            action="设计学习进展",
            approach=[
                "从基础到高级排序概念",
                "将信息分块为可管理的部分",
                "为复杂概念融入脚手架",
                "建立知识验证点"
            ]
        },
        /develop{
            action="以教学重点创建教学内容",
            sections=[
                /introduction{
                    elements=["学习目标", "与学习者的相关性", "进展概述"],
                    purpose="建立学习的动机和背景"
                },
                /core_content{
                    for_each="learning_objective",
                    structure=[
                        "概念解释",
                        "演示或示例",
                        "引导练习机会",
                        "常见陷阱和解决方案"
                    ]
                },
                /reinforcement{
                    elements=["摘要", "练习", "实际应用"],
                    purpose="巩固和应用新知识"
                }
            ]
        },
        /enhance{
            elements=[
                "视觉学习辅助(图表、图表等)",
                "不同学习风格的多样化示例",
                "复杂概念的类比和隐喻",
                "知识验证检查点"
            ]
        },
        /finalize{
            action="优化学习有效性",
            checks=["清晰度", "参与度", "知识构建", "实际应用"]
        }
    ],
    output={
        instructional_content="具有有效进展的完整以学习为重点的内容",
        quick_reference="关键概念摘要供以后审查",
        practice_materials="技能发展的练习或活动",
        instructor_notes="教学或促进指导(如果适用)"
    }
}

我想按照此协议使用我提供的信息创建教学内容。请确认并从设计学习进展开始。
```

### 实施指南

1. **主题定义**:
   - 定义要教授的具体技能或知识
   - 建立包含/排除的明确边界
   - 将复杂主题分解为可教授的组件

2. **学习者资料开发**:
   - 定义先前知识和经验水平
   - 识别学习此主题的动机
   - 考虑潜在的挑战或误解

3. **学习目标制定**:
   - 编写具体的、可测量的目标
   - 使用指示掌握水平的行动动词
   - 确保目标逐步构建

4. **格式选择**:
   - 教程:特定任务的分步指导
   - 指南:带应用的全面概述
   - 课程:结构化、多模块学习体验
   - 参考:持续使用的组织信息

### 性能指标

| 指标 | 描述 | 目标 |
|------|-------------|--------|
| 概念清晰度 | 解释的可理解性 | 目标学习者资料可访问 |
| 学习进展 | 知识的逻辑构建 | 从基础到精通的清晰路径 |
| 参与度 | 学习者兴趣的保持 | 保持注意力的多样化方法 |
| 应用支持 | 知识的实际使用 | 多个应用学习的机会 |

### 应用示例

```
/document.instructional{
    intent="创建有效的以学习为重点的内容",
    input={
        subject="使用Tableau进行数据可视化基础",
        learner_profile="具有基本数据素养但无可视化工具经验的营销专业人士",
        learning_objectives=[
            "连接并准备可视化的数据源",
            "为不同分析目的创建五种基本图表类型",
            "设计组合多个可视化的交互式仪表板",
            "应用清晰和视觉传达的最佳实践"
```

## 性能优化技巧

### Token效率

1. **内容优先级排序**:
   - 专注于影响最大的文档元素
   - 对详细信息使用渐进式披露
   - 在关键领域偏好深度而非所有领域的广度

2. **流程精简**:
   - 对于简单文档,降低流程复杂性
   - 组合相关开发步骤
   - 删除已充分理解格式的验证步骤

3. **输出焦点**:
   - 只请求需要的文档组件
   - 匹配实际使用要求的详细级别
   - 使用结构化格式进行高效信息呈现

## 结论:文档创作的演变

文档协议代表了传统内容创作方法的范式转变。通过为文档开发提供明确的架构和流程,它们将不可预测、费力密集的写作转变为结构化、可靠的内容生产。

在构建文档协议库时,请记住这些原则:

1. **从高价值文档开始**:专注于经常创建或关键的文档类型
2. **基于结果迭代**:基于实际使用结果优化协议
3. **分享和标准化**:为一致质量创建组织标准
4. **以系统思考**:考虑文档如何在更大的沟通生态系统中协同工作
5. **平衡结构和创造力**:提供足够的结构以保持一致性,同时允许适当的灵活性

有了这些原则和本指南中的文档协议,您已经具备了将内容创作从不可预测、费力密集的工作转变为可靠、高效的高质量文档生产的充分准备。

- `/enhance`:添加提高影响的元素
- `/finalize`:优化最终交付
